## 一、项目目标

将标准 KiCad Interactive BOM (iBOM) 导出的 HTML 文件，**无需修改原理图**，自动注入以下能力：

| 功能 | 说明 |
|------|------|
| **字段注入** | 在 BOM 中增加「故障点」「故障现象」两列 |
| **内联编辑** | 双击单元格弹出输入框，实时编辑 |
| **数据持久化** | 编辑结果自动保存到浏览器 `localStorage`，刷新后恢复 |
| **UI 集成** | 在 IO 菜单中增加「清除所有维修信息」按钮 |

工作流：`python patch_ibom.py <input.html>` → 生成 `Repair-Guide_<input.html>`

## 二、关键技术实现

### 2.1  注入点识别（字符串 + 正则双策略）

iBOM HTML 是单文件全量输出，所有 JS/CSS 都内联。脚本通过以下方式定位注入点：

- **Config 字段注入**：正则提取 `var config = {...}` 中的 `"fields"` 数组，用 `json.loads` 解析后追加字段，再写回。
- **CSS 注入**：`rfind('</style>')` 找最后一个 style 标签闭合处插入，避免误命中内联样式。
- **JS 函数注入**：字符串匹配 `function populateBomHeader`（稳定锚点），在其正上方插入一整块持久化函数。
- **`loadCustomFields()` 调用时机**：`rfind('initDone = true;')` 找最后一次出现，即 `window.onload` 中（而非 `overwriteSettings()` 中），确保 `storage` 和 `storagePrefix` 已初始化。
- **可编辑格子逻辑**：正则匹配 `populateBomBody` 中处理自定义字段的代码块，注入 `ondblclick` 回调。

### 2.2  localStorage 持久化机制

```
保存：saveCustomFields()
  └─ key = storagePrefix + "ibom_custom_fields_" + title + "_" + date
  └─ storage.setItem(key, JSON.stringify(pcbdata.bom.fields))

加载：loadCustomFields()（在 window.onload 末尾调用）
  └─ initMissingFields() 先将所有行补齐到 config.fields 长度
  └─ storage.getItem(key) → JSON.parse → 遍历还原故障点/故障现象
```

触发保存的时机：输入框失焦（`onblur`）或按 Enter 键，**无需手动保存**。

### 2.3  `initMissingFields()` 的作用

原始 `pcbdata.bom.fields` 只有原理图中定义的字段数目。新增的「故障点」「故障现象」不在其中，直接渲染会出现 `undefined`。  
该函数遍历所有行，将每行数组长度补齐到 `config.fields.length`，填充空字符串。

## 三、踩过的坑（按时间顺序）

### 坑 1：注入点时机错误 — `storage` 为 null

**现象**：`loadCustomFields()` 被调用，但完全没有恢复数据。  
**根因**：最初将调用注入在 `initRender()` 之后，而 `initStorage()` 在 `initRender()` 之后才运行，此时 `storage` 变量尚为 `null`，函数静默失败。  
**修复**：改用 `rfind('initDone = true;')` 找到 `window.onload` 末尾，确保 `initStorage()` 已先执行。

---

### 坑 2：`initDone = true` 出现两次

**现象**：使用 `find()` 找第一次出现时，命中的是 `overwriteSettings()` 内部的赋值，而非 `window.onload` 中的正确位置。  
**修复**：改用 `rfind()` 找最后一次出现，即 `window.onload` 中的正确时机。

---

### 坑 3：`initMissingFields()` 不能遍历对象键

**现象**：BOM 渲染出现 `undefined` 文字。  
**根因**：最初用 `Object.values(pcbdata.bom.fields).forEach(...)` 遍历，但 `pcbdata.bom.fields` 是 `{0:[...], 1:[...], ...}` 这种数字键对象，`Object.values` 在某些版本的 iBOM 中行为不符合预期（或返回顺序不稳定）。  
**修复**：改为 `for (var id in pcbdata.bom.fields)` 直接迭代键，确保每行都被正确补齐。

---

### 坑 4：`loadCustomFields()` 的 for 循环永不执行（核心 Bug）

**现象**：编辑后数据写入 localStorage 成功，刷新页面后数据丢失。  
**根因**：`pcbdata.bom.fields` 是对象（非数组），`JSON.stringify` 后再 `JSON.parse` 仍为对象，`savedFields.length` 为 `undefined`，导致：
```javascript
for (var i = 0; i < undefined; i++)  // 循环体从不执行
```
数据已保存，但加载时无法写回，表现为「保存成功、刷新丢失」。  
**修复**：将循环改为 `for (var id in savedFields)`，与 `initMissingFields` 保持一致，兼容数组和对象两种格式。

---

### 坑 5：CSS 注入命中错误的 `</style>` 标签

**现象**：CSS 被注入到页面中部某个内联 style 块末尾，导致样式失效或重复。  
**修复**：改用 `rfind('</style>')` 找最后一个闭合标签（即 iBOM 主样式块的末尾）。

---

### 坑 6：浏览器子代理无法点击真实的刷新按钮

**现象**：验证持久化时，无法在录屏中看到「点击刷新按钮」的操作。  
**根因**：Antigravity 的浏览器子代理运行在无边框嵌入式视图中，坐标系从页面内容区域开始，浏览器工具栏（地址栏、刷新按钮）不在其可操作范围内。  
**结论**：刷新验证需由用户在真实浏览器中操作，子代理只能通过 `open_browser_url` 导航同一 URL 来模拟刷新。
